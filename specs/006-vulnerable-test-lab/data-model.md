# Data Model: Vulnerable Test Lab

**Phase**: 1 — Design & Contracts
**Date**: 2026-06-02
**Status**: Final

## Entities

### VulnerableTarget

A sandboxed instance running intentionally vulnerable software. Each target has a known CVE, a predefined service endpoint, and a specific exploit mechanism.

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Short identifier (drupal, tomcat, flask, thinkphp, docker-api) |
| `cveId` | string | CVE identifier (e.g., "CVE-2018-7600") |
| `cvssScore` | number | CVSS 3.1 severity score |
| `vulhubPath` | string | Relative path in VulHub repository |
| `hostPort` | number | Port published on host machine |
| `containerPort` | number | Internal port inside the container |
| `protocol` | string | Protocol (http, tcp) |
| `requiresBuild` | boolean | Whether Docker image needs local build |
| `requiresPrivileged` | boolean | Whether container runs in privileged mode |
| `exploitType` | string | Category (web-rce, file-upload, ssti, docker-api) |
| `validationLayers` | string[] | Ordered list of validation techniques (L1, L2, L3) |

**Relationships**:
- A VulnerableTarget belongs to exactly one LabNetwork
- A VulnerableTarget has one or more ExploitPlaybook entries
- A VulnerableTarget produces zero or more ExploitReport entries

---

### LabNetwork

An isolated Docker bridge network for target communication and agent discovery.

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Network name (e.g., "targets") |
| `subnet` | string | CIDR notation (e.g., "172.20.0.0/24") |
| `driver` | string | Docker network driver ("bridge") |

**Relationships**:
- A LabNetwork contains one or more VulnerableTargets
- A LabNetwork is created by the DeployCommand

---

### DeployCommand

An orchestration command that provisions, builds, and starts all lab targets.

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | Command type (deploy, stop, reset) |
| `scriptPath` | string | Relative path to the script |
| `expectedDuration` | string | Max expected execution time |
| `preconditions` | string[] | Required conditions (e.g., "docker available") |

**Relationships**:
- A DeployCommand manages all VulnerableTargets in the LabNetwork

---

### ExploitPlaybook

Instructions for exploiting a specific vulnerable target. Used by autonomous agents.

| Field | Type | Description |
|-------|------|-------------|
| `targetName` | string | References a VulnerableTarget |
| `httpMethod` | string | HTTP method (GET, POST, PUT) |
| `endpoint` | string | URL path or API endpoint |
| `payloadTemplate` | string | Template for the exploit payload |
| `headers` | map | Required HTTP headers |
| `validationCommand` | string | Command to run post-exploitation to confirm RCE |
| `expectedOutput` | string | Pattern to match in response for success detection |

**Relationships**:
- An ExploitPlaybook is associated with exactly one VulnerableTarget
- An ExploitPlaybook produces zero or more ExploitReport entries

---

### ExploitReport

A structured report generated by an agent after attempting exploitation.

| Field | Type | Description |
|-------|------|-------------|
| `agentId` | string | Unique agent identifier |
| `targetName` | string | The target that was exploited |
| `cveId` | string | CVE matched |
| `timestamp` | datetime | When exploitation occurred |
| `success` | boolean | Whether exploitation was confirmed |
| `validationLayersUsed` | string[] | Which validation layers produced results |
| `validationOutput` | map | Output from each validation layer |
| `commandExecuted` | string | The exact command/request sent |
| `responseSummary` | string | Truncated response for audit |

**Relationships**:
- An ExploitReport references one VulnerableTarget
- An ExploitReport references one ExploitPlaybook
- An ExploitReport is sent to the central API (POST /api/reports)

---

## State Transitions

### Lab Lifecycle

```
[Not Deployed] → deploy → [Running: All Targets Up] → stop → [Stopped]
                                                           ↓
                                                     reset → [Not Deployed] → deploy → [Running: Clean State]
```

### Target Lifecycle

```
[Building] → build success → [Running] → exploit attempt → [Compromised] → reset → [Clean: Rebuilt]
                 ↓ fail                          ↓ fail
           [Build Failed]                   [Still Running]
```

## Validation Rules

From spec requirements:

- **FR-006**: Deploy command must start all 5 targets within 2 minutes
- **FR-009**: All targets on same isolated network, fixed addresses
- **FR-010**: All targets reachable from host via localhost
- **FR-011**: 3-layer validation (L1→L2→L3 fallback), no whoami dependency
