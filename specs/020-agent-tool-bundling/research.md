# Phase 0 Research: Agent Tool Bundling

## 1. How to embed non-Java binary resources in a GraalVM native image

**Decision**: Store tool binaries as plain classpath resources under `src/main/resources/tools/<os>-<arch>/<tool>`, and register them for native-image inclusion via a Spring AOT `RuntimeHintsRegistrar` (`hints.resources().registerPattern("tools/**")`), the same mechanism Spring Boot uses to make arbitrary classpath resources reachable in a native image without reflection.

**Rationale**: `agents/unix` already proves classpath resources survive native-image compilation and stay accessible at runtime — `install-agent-http.sh.tmpl` and `install-agent-transfer.sh.tmpl` are loaded via `ClassPathResource` in the shipped native binary today (per `AGENTS.md`'s script/template boundary rule). The only gap for binary (non-text) resources is that GraalVM's static analysis won't auto-detect resource access patterns that don't look like typical `ClassPathResource` reads; an explicit `RuntimeHintsRegistrar` closes that gap without reflection.

**Alternatives considered**:
- *Native-image `resource-config.json` maintained by hand* — works, but Spring's `RuntimeHintsRegistrar` is the project-idiomatic, framework-managed equivalent already implied by using Spring Boot 3, and keeps the hint co-located with the Java code that needs it.
- *Ship tool binaries as a separate tarball alongside the agent executable* — rejected because the existing install flow (`install-agent-http.sh.tmpl`) downloads and runs a single raw binary file (`[binary bytes]\n{manifest json}`), with no mechanism today to also stage a sibling directory of files next to it; changing that download/verification protocol would be a much larger change than embedding resources in the binary that already exists.

## 2. Selecting the right binary variant for the running host

**Decision**: Organize bundled binaries in per `os.name`-`os.arch` subdirectories (`linux-amd64/`, `darwin-arm64/`) inside `tools/`; `BundledToolProvisioner` reads `System.getProperty("os.name")` / `System.getProperty("os.arch")` at startup, normalizes them (e.g. `Mac OS X` + `aarch64` → `darwin-arm64`), and extracts only the matching subdirectory's files.

**Rationale**: The native agent binary itself is already architecture-specific — `package-macos.sh` compiles for the local macOS/arm build, and production deployment targets Linux amd64 per the existing `019-expand-remediation-strategies` plan's Technical Context. Embedding all known variants as classpath resources (not per-arch Maven profiles) keeps the build script surface unchanged (`./mvnw -Pnative native:compile ...` stays the only build command) at the cost of a bounded amount of unused resource weight in any single compiled binary — an acceptable tradeoff for a prototype/thesis-scope agent with a small, fixed capability set (4 tools).

**Alternatives considered**: Per-arch Maven resource-filtering profiles that only bundle the matching variant at build time — rejected for now because there is no existing multi-arch build/CI pipeline in the repo to select the right profile automatically; introducing one is out of scope for this feature and would violate the "smallest correct change" principle.

## 3. Making bundled tools usable both by dedicated step handlers and by free-form scripts

**Decision**: At startup, `BundledToolProvisioner` extracts the matched binaries to a runtime temp directory (`Files.createTempDirectory("agent-tools")`), sets the executable bit on each file, and exposes the directory's absolute path. `CommandExecutor`'s `ProcessBuilder` is updated to prepend this directory to the `PATH` environment variable it passes to every command it runs.

**Rationale**: Not all tool invocations go through a dedicated `StepHandler` — `ExecuteExploitStepHandler` and `RemediationStepHandler` both execute arbitrary command strings that originate from the central platform (`EXPLOITATION_KNOWLEDGE` scripts, remediation strategy commands) and may already reference `nmap`, `nc`, `curl`, etc. by bare name, assuming they're on `PATH`. Prepending the extraction directory to `PATH` once, centrally, makes every command executed by the agent (handler-driven or script-driven) able to resolve these tools by name, without hardcoding extracted paths into every call site.

**Alternatives considered**: Resolve and pass full absolute paths to each tool invocation individually — rejected because it would require every future script/strategy author (including content generated by the central platform's `EXPLOITATION_KNOWLEDGE`/remediation catalog) to know the agent's internal extraction directory layout, which is neither stable nor part of any documented contract.

## 4. Tool licensing and distribution

**Decision**: Bundle statically-linked builds of nmap, rustscan, and a BSD/MIT-licensed `nc` variant (e.g. `netcat-openbsd` or `ncat` depending on final build availability) as specified in the user's technical view; use `curl` (permissive MIT/X-derivative license) rather than `wget` (GPL) for the file-retrieval capability, since `curl`'s license imposes materially lighter redistribution obligations for a bundled-binary distribution model.

**Rationale**: nmap is GPLv2; rustscan is GPLv3. Bundling GPL-licensed binaries inside a redistributed product carries source-availability obligations under those licenses. This is a known, documented tradeoff rather than a blocker: the user's technical view explicitly named nmap and rustscan, and this is an academic/prototype (TFG) system, not a commercial redistribution — the licensing implication is recorded here so it is a conscious, visible decision rather than an oversight, and is flagged as a Constitution/assumption note in `spec.md`. If the project moves toward wider redistribution later, replacing nmap/rustscan with permissively-licensed or custom-built equivalents (e.g. a minimal Go-based scanner) is a follow-up, not part of this feature's scope.

**Alternatives considered**: Custom-written minimal scanner (e.g. a small Go/Rust static binary implementing TCP connect scans) to avoid GPL entirely — rejected as out of scope for this feature; the user's technical view specifically asked for these named tools, and building/maintaining replacements is a materially larger effort than this feature's bounded goal (make the agent self-contained using the tools already chosen).

## 5. Timeout and failure reporting for bundled-tool invocations

**Decision**: Reuse the existing `CommandExecutor` contract as-is — it already enforces a caller-supplied timeout via `Process.waitFor(timeoutSeconds, ...)` and force-destroys the process on timeout, returning `TaskResult.failure(...)`. New scan/fetch handlers call `commandExecutor.execute(command, timeoutSeconds)` exactly like `RemediationStepHandler`/`ExecuteExploitStepHandler` do today, and map a failed `TaskResult` (including timeout) to `StepResult.failure(action, logs)` with the underlying failure reason included in the logs.

**Rationale**: This is already the project's established pattern (`RemediationStepHandler.executeCommands`, `ExecuteExploitStepHandler.handle`); no new timeout/process-management code is needed, satisfying FR-005 and FR-009 with zero new infrastructure.

**Alternatives considered**: A dedicated timeout wrapper per bundled tool — rejected as unnecessary duplication of behavior `CommandExecutor` already provides uniformly.

## 6. Threading the plan's target address into step handlers

**Decision**: Add a `String targetIp` parameter to `StepHandler.handle(StepAction action, Map<StepAction, StepResult> context, String targetIp)`; `WorkerCoordinator.runJob()` passes `planResponse.getTargetIp()` into `taskExecutionService.executeJob(jobId, steps, actions, targetIp)`, which threads it into every `handler.handle(...)` call.

**Rationale**: Confirmed by reading the current code: `WorkerCoordinator` already fetches `planResponse.getTargetIp()` (`WorkerCoordinator.java:74`) but stores it only in a local instance field that no `StepHandler` receives — `TaskExecutionService.executeJob` builds its `context` map from `Map<StepAction, StepResult>` only, and no existing handler has a way to learn the scan target. `RemediationStepHandler` currently works around a related gap by scraping a `"targetId:"`-prefixed log line out of the `SERVICE_SCAN` step's own logs — but today `SERVICE_SCAN` is `EchoStepHandler`, which never emits that line, so this path is effectively dead until real scan handlers exist. This feature is the natural point to fix the gap properly rather than propagate another log-scraping convention.

**Alternatives considered**: Keep scraping `targetId:`-prefixed log lines from a predecessor step's `StepResult.getLogs()` (the existing `RemediationStepHandler` convention) — rejected as the smaller-looking but actually worse option: it depends on step ordering and string-prefix conventions instead of an explicit parameter, is what caused the current dead-path bug, and the constitution's "smallest correct change" principle favors fixing the root cause over extending a fragile pattern into two more handlers.
